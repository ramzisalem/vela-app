# 37 — Diary Layer

## Why this exists

Faces don't change in a vacuum. They change because someone slept badly, started a new product, got a sunburn, broke up, traveled, was sick. Without that context, every chart in Vela is data without explanation. With it, the same chart becomes a record the user can actually learn from.

The diary is also the layer that lets every other Vela feature behave intelligently:

- **Correlation engine (file 33)** can read diary tags to exclude unrepresentative weeks.
- **Aging band (file 36)** can suppress callouts when a logged event explains a sharp move.
- **Treatment tracking (file 34)** can attach side-effect logs to a richer journal entry.
- **Hair tracking (file 35)** can defer alarm on postpartum or hat-week scans.
- **Monthly Wrapped (file 38)** can quote the user back to themselves from their own words.

Done well, the diary is the most beloved feature in the app — the one users tell other people about. Done poorly, it's an empty text box nobody touches.

This file extends `02_TYPES_AND_MODELS.md`, `03_BACKEND_SUPABASE.md`, `06_AI_PROMPTS.md`, `09_ROUTINE.md`, `10_DASHBOARD.md`, `12_NOTIFICATIONS.md`, `14_SETTINGS_AND_SUBSCRIPTION.md`, `20_INFORMATION_ARCHITECTURE.md`, `25_ANALYTICS.md`, and `37_DIARY.md` is referenced by 33, 34, 35, 36, 38.

## Product principles

1. **Friction is the enemy.** Logging an entry has to feel as light as sending a one-line text. Three taps to journal, end to end, on the median path.
2. **Optional, always.** A user who never opens the diary still gets a great app. Nothing about the UX should imply they're failing if they don't write.
3. **Plain and warm.** Not "Reflection." Not "Wellness Journal." Not "How are you feeling today?" — *Add a note* is the language. Calm, direct.
4. **Auto-tagging is invisible by default.** Users see their own words in their own voice. Tags exist behind the scenes to make other features smarter; they surface only on demand.
5. **Encrypted at rest, never used to train.** This is the most personal data in the app. The privacy story is non-negotiable.

## Types

```ts
// src/types/diary.ts
// Add to 02_TYPES_AND_MODELS.md.

export interface DiaryEntry {
  id: string;
  userId: string;
  // The "subject" of the entry. A diary entry can attach to:
  //   - a date (general daily journaling),
  //   - a specific scan (notes about that scan),
  //   - a treatment (treatment-specific reflections).
  attachedTo:
    | { kind: 'date'; date: string }                   // YYYY-MM-DD
    | { kind: 'scan'; sessionId: string }
    | { kind: 'treatment'; treatmentId: string };
  // The content. Either typed or transcribed.
  body: string;                                          // free text, 1–5000 chars
  // Optional structured tags chosen by the user.
  userTags: DiaryUserTag[];
  // Tags inferred by the AI proxy from the body. Hidden in the UI by default.
  inferredTags: DiaryInferredTag[];
  // The user can mark this entry as a "skip from analysis" signal —
  // tells the correlation engine and aging band to ignore the relevant week.
  excludeFromAnalysis: boolean;
  // Voice-to-text origin if relevant.
  source: 'typed' | 'voice';
  createdAt: string;
  updatedAt: string;
}

// User-chosen, structured chips. Limited set so they're useful for analysis.
export type DiaryUserTag =
  // Sleep & stress
  | 'slept-poorly' | 'slept-well' | 'stressed' | 'rested'
  // Body & cycle
  | 'sick' | 'period' | 'pms' | 'pregnant' | 'postpartum' | 'menopause'
  // Lifestyle
  | 'travel' | 'sun' | 'alcohol' | 'late-night' | 'workout-hard'
  // Hair-specific (file 35)
  | 'haircut' | 'hat-day' | 'hair-wet' | 'hair-styled'
  // Skin-specific
  | 'sunburn' | 'breakout' | 'flare-up' | 'allergic-reaction'
  // Products & treatments
  | 'new-product' | 'stopped-product' | 'changed-routine'
  // Life
  | 'big-life-event' | 'good-day' | 'rough-day';

// Tags inferred by AI; same vocabulary as user tags + the diary engine
// can suggest "promote to user-tag" if it sees the same inference repeatedly.
export type DiaryInferredTag = DiaryUserTag;

export interface DiaryWeeklySummary {
  // Generated by the AI proxy weekly when there are entries.
  weekIso: string;                                       // e.g., "2026-W18"
  userId: string;
  themes: { tag: DiaryInferredTag; count: number }[];
  oneLineSummary: string;                                // ≤120 chars
  generatedAt: string;
}
```

### Why structured tags AND free text?
- **Free text** captures the user's voice, lets them say things tags can't (a sentence about how they felt going into a meeting, what their dad said about their hair).
- **Structured tags** make the data machine-readable for the correlation engine without forcing AI inference on every entry. Quick chips at the bottom of the entry sheet let users tag in one tap.

We support both. The user picks zero or many tags; AI fills in inferred tags after the fact for analytics use.

## The entry sheet

Bottom sheet, opens from many entry points (see "Where the diary lives" below). Designed to take ≤8 seconds for a one-line entry.

```
┌─────────────────────────────────────────────┐
│         Add a note · Tuesday, May 7         │
├─────────────────────────────────────────────┤
│                                             │
│  ┌───────────────────────────────────────┐  │
│  │  Type or tap the mic.                 │  │
│  │  (large soft text input)              │  │
│  │                                       │  │
│  └───────────────────────────────────────┘  │
│                                             │
│  Tag this if you want · scrollable chips    │
│  ╭──────────╮ ╭──────────╮ ╭──────────╮     │
│  │ Slept    │ │ Stressed │ │ Travel   │ ... │
│  │ poorly   │ │          │ │          │     │
│  ╰──────────╯ ╰──────────╯ ╰──────────╯     │
│                                             │
│  ☐ Don't include this week in patterns      │
│                                             │
│  ┌──────────────────┐  ┌──────────────────┐ │
│  │       Mic        │  │      Save        │ │
│  └──────────────────┘  └──────────────────┘ │
└─────────────────────────────────────────────┘
```

- **Date subject**: defaults to today, can change with a small calendar tap on the title.
- **Text input**: serif, body-sized, large hit target. Multi-line auto-grow.
- **Mic button**: invokes Apple's on-device speech-to-text (`SFSpeechRecognizer`, on-device locale). The transcribed text drops into the input where the user can edit before saving. If `SFSpeechRecognizer.supportsOnDeviceRecognition` is `false` for the user's locale (some non-English locales), the mic button is rendered greyed-out with the tooltip *"Mic available when your device supports it."* No error sheet, no nag.
- **Tag chips**: horizontally scrollable, multi-select. The order is contextual — "Slept poorly" / "Stressed" surface first if it's morning; "New product" / "Sunburn" surface first if the routine engine detected a recent change.
- **Exclude-week toggle**: small, easy to find. Tapping it doesn't trigger any visual confirmation — the toggle state IS the confirmation.
- **Save** commits the entry to local DB immediately; sync to Supabase happens in background.

### Voice-to-text rules

- All speech recognition runs on-device. The audio buffer is never sent to a server. (`requiresOnDeviceRecognition = true`.)
- The mic button shows a soft pulsing waveform while listening and stops when the user pauses for >1.5s or taps it again.
- Transcripts are editable before save. We never auto-commit a transcription.
- The `source` field on the entry tracks origin for analytics, never surfaced to the user.

## Where the diary lives

### Primary entry points
1. **Daily nudge** — at the user's chosen time (defaults to 9pm), an opt-in notification: *"Add a note for today?"* Single tap opens the entry sheet.
2. **Post-scan reveal screen** — a small "Add a note about this scan" link below the score. Pre-attaches the new entry to the scan.
3. **Routine screen** — a "How was today?" prompt at the bottom of the daily routine card. Visible once per day.
4. **Treatment timeline (file 34)** — every treatment has its own diary stream attached.
5. **History tab** — a calendar view; tap any date to read or add an entry.
6. **Quick action from the iOS share sheet** (v1.5) — share a thought from anywhere into a Vela diary entry.

### Diary tab placement (IA)
A new top-level tab? No. The diary lives:
- As a section *inside* the History tab in the IA tree (file 20).
- As contextual entry sheets accessed from the dashboard, routine, scan reveal, and treatments.

This avoids cluttering the bottom tab bar and keeps the diary feeling like a natural extension of the data, not a separate journaling app.

### History view
Calendar grid (familiar weekly layout, soft cream cells), each day shows:
- A small dot if a scan exists.
- A small line-icon if a diary entry exists.
- A subtle treatment-color dot if a treatment was active that day.

Tap a day → bottom sheet shows that day's scan summary, diary entries, and treatments. Above-fold scrolling-friendly.

## AI inference & weekly summary

### Auto-tagging
After save, an Edge Function processes the entry asynchronously:

1. The body text is sent to the AI proxy (file 06) with `DIARY_INFERENCE_SYSTEM`.
2. The AI proxy returns a list of `DiaryInferredTag` values from the controlled vocabulary.
3. Tags are stored on the entry and used by other features (correlation engine, aging band).
4. The user does not see the inferred tags by default. They appear in the diary entry detail view under a *"Tags Vela noticed"* expandable.

#### `DIARY_INFERENCE_SYSTEM` prompt

```
You read a short diary entry and return any tags from this fixed list that the
text clearly implies. You do not invent tags. You do not return tags you are not
confident about.

Available tags:
[list of DiaryUserTag values]

VOICE: this is a backend tagger; output is a JSON array of strings only.

Rules:
- Be conservative. If the entry is "Long day," return [].
- Tag based on what the user wrote, not what it might mean. "Couldn't sleep"
  returns ["slept-poorly"]. "Stressed about Mom" returns ["stressed"], not
  ["big-life-event"] — there's not enough context.
- Return at most 4 tags per entry.
- The "exclude-from-analysis" tags (sick, postpartum, big-life-event) are
  high-bar — only return when explicitly stated.
- LIFE-STAGE MODE TAGS — STRICT RULE (file 48 boundary):
  Do NOT infer "pregnant", "postpartum", "menopause" from context alone.
  Only return these tags when the user EXPLICITLY typed those words.
  "Tired all the time" is NOT pregnancy. "Hot flashes" is NOT menopause unless
  the user used that word. The user must opt INTO life-stage modes via Settings;
  diary AI never pre-empts that choice.

Return ONLY a JSON array. No other text.
```

### Weekly summary

Once a week (Sunday evening), if the user has at least three entries that week, a `DiaryWeeklySummary` is generated:

- Top 3 inferred tags with counts.
- One-line summary in the user's voice register, ≤120 chars.

Surfaced in:
- Dashboard *"Patterns we noticed"* card (file 33), if relevant.
- Monthly Wrapped (file 38).
- The Sunday-evening notification: *"Your week in three words: rough sleep, two great workouts, sun on Saturday."*

The summary copy is intentionally a list-of-fragments style, not prose. Reads like a friend's text recap, not a wellness app.

#### `DIARY_WEEKLY_SUMMARY_SYSTEM`

```
You receive a week's worth of diary entries from one user. You write ONE line,
≤120 characters, summarizing the week as a list of fragments separated by commas.

VOICE: Vela voice. Plainly factual. Not therapeutic.

Rules:
- Quote the user's actual words when possible.
- Never interpret or moralize ("you should rest").
- Never use the word "self" or "wellness."
- 2–4 fragments, comma-separated.

EXAMPLES — good:
- "Two rough nights, presentation Wednesday, glass of wine Saturday."
- "Sunny week in Lisbon, lots of walking, slept hard."
- "Period started, broke out a little, otherwise quiet."

EXAMPLES — bad:
- "Your wellness journey shows mixed results this week."
- "You experienced stress and you should consider self-care."
```

## Privacy

The diary is the most sensitive data Vela holds. The privacy treatment is stricter than for face data:

### Storage
- All diary entries are encrypted at rest in Supabase using a per-user key stored in the user's iCloud Keychain. The Supabase row stores ciphertext only.
- Local WatermelonDB stores plaintext (encrypted by iOS file system protection, like all on-device user data).
- Inferred tags ARE stored on the row in plaintext (they're a short controlled vocabulary). The body text is the field that's encrypted.

### Why client-side encryption for the body
- A Supabase admin (or attacker who breaches the database) cannot read user diary text.
- The trade-off: server-side full-text search and server-side AI processing of the body becomes impossible. We accept this; AI processing happens on the user's device or via Edge Function with the body briefly decrypted in memory and never logged.
- Account deletion (file 14) drops the encrypted blob; loss of the keychain key is treated as data loss the same way (the user is told this once during diary onboarding).

### Edge Function handling
- The AI proxy (file 06) receives diary text only when generating inferred tags or weekly summaries. Each call is logged with metadata (user, action, timestamp) but **never** the body or response. Sentry `beforeSend` is extended to scrub any field named `body`, `entry`, `text`, `summary`, or `transcription`.
- OpenAI's API usage policies apply to these calls (verify current enterprise / API data handling before launch).

### Export
- Data export (file 14) includes diary entries as a JSON file. The export is a one-time decryption operation; the user is told the export contains their journal text and that they should treat the file as sensitive.

### Onboarding privacy primer (one-time, on first diary use)

> **Just so you know**
>
> Your diary entries are encrypted on your phone with a key only you have. Vela's servers can't read them. We use Apple's on-device speech recognition for the mic, so your voice never leaves your phone either.
>
> If you delete your account, your diary goes with it. If you lose access to your iCloud Keychain, we won't be able to help you recover them — that's a privacy trade-off we made on purpose.
>
> *[ Got it ]*

## Notifications

A new notification kind in file 12: `diary-nudge`.

- Default: opt-in via "Daily diary nudge" Settings toggle, off by default. (Anti-pattern note: never default this on.)
- Time: user-configurable, default 9pm local time. Skip on days the user already wrote.
- Copy: *"Add a note for today?"* Tap deep-links to the entry sheet pre-set to today's date.

All diary notifications respect Do Not Disturb and the user's overall notification preferences.

## Edge cases

- **User writes a very long entry (5000+ chars)** — the input cap is 5000; UI shows characters remaining when within the last 200. Above 5000 the user is prompted to break the entry into two.
- **User edits an entry days later** — `updatedAt` updates; the inferred tags are re-run. Original `createdAt` is preserved for the analysis attribution (the entry counts toward the day it was originally about, not the day of edit).
- **User deletes an entry** — soft delete in WatermelonDB (set `deletedAt`); hard delete from Supabase ciphertext blob immediately. Inferred tags are removed from analytics.
- **Voice transcription is wrong** — user edits text before saving. We never log the transcription mistake.
- **Speech recognition unavailable** (locale not supported, microphone permission denied) — the mic button is hidden with a small footnote *"Mic available when your device supports it."* No nag.
- **Entry attached to a scan that's later deleted** — the entry is preserved with its body and tags, but `attachedTo` falls back to a `date` reference using the scan's date.
- **Massive backfill** (user logs 30 entries in one day for past weeks) — entries with `attachedTo.date` in the past are accepted; the inferred-tag pipeline handles them in a queue, not synchronously.
- **Diary nudge fires while user is mid-app** — silent in-app banner instead of system notification, dismissible.
- **User opts out of the weekly summary mid-week** — the in-progress summary is dropped. The toggle has no effect on tags or correlations, only on the surfaced summary.

## Settings

A new section: **Settings → Diary**.

- **Daily nudge** — toggle, with time picker. Default off.
- **Sunday weekly summary** — toggle. Default off.
- **Voice notes** — toggle (defaults to on; turning off hides the mic button).
- **Show inferred tags on entries** — toggle. Default off (they're invisible by default).
- **Allow Vela's AI to read entries for tagging and weekly summary** — toggle. Default on. **If turned off, all AI-derived features that depend on diary content (correlations, weekly summary, tag inference) gracefully degrade.** A footnote explains this.
