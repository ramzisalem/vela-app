# 40 — Pre-Paywall Value Layer

## Why this exists

The single most leveraged moment in any subscription app is the few minutes between sign-up and the paywall. Every user who completes onboarding but bounces at the paywall is a user we already paid an acquisition cost for and got nothing from. The fix isn't more aggressive friction — it's giving the user *something tangible* before we ask them to pay, so the paywall feels like an extension of value rather than a sudden gate.

This file specifies two new pre-paywall surfaces and a friction audit for the existing onboarding flow:

1. **A "your face right now" preview card** — the user's first scan, summarized in one personalized observation, shown right before the paywall.
2. **A personalized routine preview card** — three to four task names that *will* be in their generated routine, shown right after the face preview.
3. **An onboarding friction audit** — every screen between sign-up and the first scan reveal must earn its place by changing what the user sees on the other side. This file makes the rule explicit and provides the audit framework.

The result: a user who hits the paywall has already received something. Their face has been observed. Their routine has been previewed. They feel seen. The paywall becomes "subscribe to keep this going" rather than "subscribe to find out if this works."

This file extends `05_CAPTURE_FLOW.md`, `06_AI_PROMPTS.md`, `07_ONBOARDING.md`, `08_PAYWALL.md`, `15_DESIGN_SYSTEM.md`, `19_USER_JOURNEYS.md`, `21_BRAND_SYSTEM.md`, and `25_ANALYTICS.md`.

---

## Product principles

1. **Generosity converts better than scarcity.** A user who receives value voluntarily is a user who pays voluntarily. Vela's brand is built on dignity; conversion mechanics that contradict that brand will hurt LTV even if they lift trial-start rates.
2. **The preview is real, not a tease.** What the user sees on the preview card actually reflects their face and their routine. We don't show generic "your skin will glow" mockups.
3. **Honesty about the AI.** The face-preview observation is one line of plain language. We don't pretend the AI saw something it didn't, and we don't oversell what one scan can tell us.
4. **Time is the constraint.** Median time from sign-up to the first preview card must be under 4 minutes. Every onboarding screen above that line gets cut.

## The new flow, end-to-end

The existing flow (file 07 onboarding):

```
Sign up → Q1 (gender) → Q1b (framework, conditional) → 28 more questions
→ Camera permission → First scan → Reveal screen → Paywall
```

The new flow (this file):

```
Sign up → Q1 (gender) → Q1b → 28 more questions
→ Camera permission → First scan → Reveal screen
→ NEW: "Your face right now" preview card
→ NEW: "Your routine, previewed" card
→ Paywall
```

The two new cards live between the existing reveal screen and the existing paywall. They are full-bleed, swipeable, and skippable (a small "Continue" tap-target advances at any time).

## Card 1 — "Your face right now"

### Visual

Full-bleed cream surface. Serif headline at top. The user's just-captured baseline scan as a softly cropped, lighting-normalized portrait below the headline. One single body line below the photo. A continue affordance at the bottom.

```
┌─────────────────────────────────────────┐
│                                         │
│   Your face,                            │
│   right now.                            │
│                                         │
│   ┌────────────────────────────────┐    │
│   │                                │    │
│   │    [user's baseline scan,      │    │
│   │     softly cropped]            │    │
│   │                                │    │
│   └────────────────────────────────┘    │
│                                         │
│   Your skin reads slightly red around   │
│   the cheeks. Worth tracking.           │
│                                         │
│                              [Continue] │
└─────────────────────────────────────────┘
```

### Typography & color

- Headline: `displaySerif` 36/40 (file 15), espresso text, two lines max.
- Body line: `bodyLarge` 18/26, `text.primary`, max 22 words.
- Photo: rounded corners (`Radii.large`), aspect ratio 4:5, gentle drop shadow respecting the elevation rules.
- Cream surface (`surface.primary`); single accent dot from `VelaPrimarySoft` palette near the bottom-right corner.
- Continue link: `text.secondary`, plain text, no button shape.

### Motion

- Card fades in over 400ms once the reveal screen completes.
- The body line types in with a 16ms-per-character animation (capped at 800ms total). Reduce-motion: line appears instantly.
- No celebratory animation. No confetti. The intent is recognition, not reward.

### Body line — what we say

Generated by the AI proxy based on the baseline scan's metrics. Always:
- One sentence.
- One observation.
- Plain language.
- Worth-noting framing, never alarming, never flattering.
- Closes with "worth tracking" or equivalent if the observation is neutral, or simply states the fact if it's clearly positive.

Examples (good):
- *"Your skin reads slightly red around the cheeks. Worth tracking."*
- *"Even tone across most of your face. We'll watch how it holds."*
- *"A little hollow under your eyes — common at this hour. Worth seeing how it looks across a week."*
- *"Your jawline reads sharp in this scan. We'll see how it tracks."*

Examples (forbidden):
- *"Your skin is glowing!"* (forbidden words from file 21)
- *"You have signs of aging."* (judgmental + medical-adjacent)
- *"Score: 73."* (we hold the number for the dashboard, not for this card)
- *"You should consider..."* (we don't recommend before paywall)

### `PREPAYWALL_FACE_PREVIEW_SYSTEM` prompt

Add to `06_AI_PROMPTS.md`:

```
You write a SINGLE plain-language sentence describing what you observe about a face
in a baseline scan. This is the user's first interaction with Vela's AI.

CONTEXT YOU WILL RECEIVE:
- Sub-score values for the baseline scan (clarity, redness, eyeArea, cheekVolume,
  jawDefinition, symmetry — file 02 types).
- Capture3D quality metrics (file 32).
- Capture conditions (time-of-day, lighting band).

VOICE: Vela voice (file 21). No exclamation marks. No forbidden words.

OUTPUT:
- Exactly one sentence, ≤22 words.
- Pick ONE observation. Do not list.
- If the observation is neutral or mixed, close with "Worth tracking." or
  equivalent ("We'll watch how it tracks", "We'll see how this holds.")
- If the observation is clearly something the user might enjoy hearing
  (e.g., even tone, sharp jawline at the moment of capture), state it plainly
  WITHOUT compliment energy. We acknowledge, we don't praise.
- If quality flags suggest poor capture conditions (low light, partial occlusion),
  bias toward "We'll get a clearer read next time" framing rather than over-claiming.

NEVER:
- Recommend a product, treatment, or routine task here.
- Use a number or score.
- Use the word "your skin's amazing", "you look great", "looking your best", etc.
- Diagnose or imply medical condition.
```

User message:
```
Sub-scores: clarity={x}, redness={x}, eyeArea={x}, cheekVolume={x},
            jawDefinition={x}, symmetry={x} (each 0-100 normalized)
Capture quality: lighting={good|low|high-contrast}, occlusion={pct},
                 time-of-day={morning|afternoon|evening}
```

### Edge cases

- **Quality too low for an honest observation** — if `lightingBand === 'low'` and `occlusionPct < 80`, skip the AI call entirely and use a deterministic line: *"This was a tricky scan to read. Your real baseline starts on the next clean one — no rush."* The card still shows; the photo is still the user's.
- **No 3D data (older device)** — the prompt receives only 2D sub-scores. Output is unchanged in style.
- **AI proxy fails** — fallback line: *"You're scanned. We'll start tracking from here."* Card still shows. We do not block the paywall on a failed AI call.
- **User dismisses the card with the Continue link** — no penalty. Their observation is generated, persisted, and shown again on the dashboard's first-scan welcome state.

## Card 2 — "Your routine, previewed"

### Visual

Same surface treatment. Different content.

```
┌─────────────────────────────────────────┐
│                                         │
│   Your routine,                         │
│   previewed.                            │
│                                         │
│   Based on what you told us, here's     │
│   what we'll start you on:              │
│                                         │
│   ─────────────────────────────────     │
│   Morning                               │
│   ✿  Gentle cleanser                    │
│   ✿  Niacinamide serum                  │
│   ✿  Mineral SPF                        │
│   ─────────────────────────────────     │
│   Evening                               │
│   ✿  Tretinoin (every other night)      │
│   ✿  Ceramide moisturizer               │
│   ─────────────────────────────────     │
│                                         │
│   You'll see the full version inside.   │
│                                         │
│                              [Continue] │
└─────────────────────────────────────────┘
```

### What's shown

Three to five tasks total, grouped by morning/evening. **Real task names from the routine the engine generated**, not placeholders. The full routine — frequencies, contraindications, evidence levels, swap groups — is held back for the post-paywall experience. The user sees what they'll be doing, not the full reasoning.

### Why this works

A routine list is the most legible "here's what I get for $79" the app can show. Skincare apps are often mocked for being vague about what subscribers actually receive; we make it concrete. The bias toward 3–5 task names (rather than 8) protects the post-paywall experience by leaving real new content there.

### Edge cases

- **Routine engine hasn't finished generating yet** — show a soft skeleton state for ≤ 2 seconds with the same headline. If it still isn't ready, skip this card entirely and proceed to paywall. Do not block.
- **Engine returns a fallback (non-AI) routine** — same surface; the user can't tell the difference, and the routine they see post-paywall will be regenerated fresh.
- **User has medical contraindications** — tasks involving contraindicated actives are filtered before the preview is rendered. The preview honors the user's onboarding answers; users who flagged "pregnant" never see retinoid tasks here.

### Skipping ahead

A small affordance below the routine list: *"Skip preview · Show me the paywall."* Tappable, no friction. ~5% of users will use it. Not a problem.

## Onboarding friction audit

The existing onboarding (file 07) has 30 questions across five sections. With the new previews, the flow is now:

```
Sign up (3 steps: email, password, name)
+ 30 onboarding questions (across 5 sections)
+ Q1b conditional reveal
+ Camera permission ask
+ First scan (alignment + capture, ~90s)
+ Reveal screen (~5s)
+ Face preview card (~10s)
+ Routine preview card (~10s)
+ Paywall
```

Median time today: ~6 minutes. Target: ≤4 minutes.

### Audit rules

1. **Every onboarding question must change something the user sees post-onboarding.** If a question's answer doesn't affect their routine, their score weights, their notification copy, or any other surfaced output, it's vanity data and gets cut.
2. **Multi-select questions are preferred over open-ended ones.** Tap targets > typing.
3. **Conditional reveals reduce visible question count without losing data.** Questions only relevant to a subset of users (e.g., menopause-related questions) reveal only when prior answers indicate.
4. **The first three questions must produce a tangible promise.** After Q3, the user should already see a single line of micro-payoff: *"OK — for someone in their 30s tracking texture, we'll bias your routine toward retinoids and gentle exfoliation. Couple more questions."*
5. **Free-text fields are last-resort.** Each one extends the median time by 20+ seconds.

### What gets cut after the audit

These are the questions in the current 30 that haven't earned their place. Cut them or move to in-app progressive disclosure:

- *"What's your zip code?"* — used only for sun-strength regional adjustment, which can be inferred from device locale.
- *"Do you wear makeup?"* — affects nothing in the routine engine; if it later becomes relevant for capture-quality nudging, ask in the diary.
- *"What's your skincare budget?"* — Vela isn't a product recommender at v1.
- *"How would you describe your skin?"* (free text) — covered by the structured skin-type question; the free text is duplicate.
- *"How important is skincare to you on a scale of 1–10?"* — vanity metric.

After cuts, target onboarding question count: 22 (with up to 4 conditional reveals = 26 max for any single user). Median time-to-paywall: ≤4 minutes.

### Micro-payoff rule

Every 5–7 questions, surface a one-line acknowledgment:

- After Q5: *"OK — for someone in their 30s tracking texture, we'll bias your routine toward retinoids and gentle exfoliation. Couple more questions."*
- After Q12: *"Sleep and stress matter for what we'll surface. We'll watch for patterns there."*
- After Q19: *"Almost done. Last few are about how you'd like to be reminded."*
- After Q26: *"That's everything. Camera time, then we're set."*

These already exist as a concept in the brand system (file 21); this file makes them mandatory and specifies cadence.

## Routine preview lock (canonical)

The routine preview card shows 3–5 task names. Once shown, those task names are LOCKED as the user's seed routine for the post-paywall experience. The routine engine does not regenerate between preview and post-paywall. This prevents the user from seeing "we'll start you on niacinamide and SPF" and then opening the dashboard to find azelaic acid and ceramide moisturizer.

The locked task list is stored as `profile.flags.previewedRoutineTaskIds: string[]` and read by the routine engine on first post-paywall load. Subsequent weekly adaptations (file 09) can change the task set freely, but the very first dashboard view matches the preview exactly.

## Reveals + micro-payoff collision

Onboarding micro-payoff cards fire after Q5/Q12/Q19/Q26. The first dashboard view post-paywall could ALSO trigger a file 43 feature-reveal card (e.g., the Apple Health Vital ask at week 2). To prevent two "new things" from competing for attention on the same first session:

- **File 43 reveal evaluator MUST suppress all reveals during the user's first post-paywall session.** First reveal eligibility starts on session 2 (i.e., the day after first dashboard view).
- This is a hard rule, not a heuristic. CI tests verify a fresh user's session 1 has zero reveal cards.

## Paywall arriving-from-preview line

The paywall (file 08) renders one of three contextual lines depending on entry path. The preview-lead-in line is gated:

```typescript
const paywallContextLine =
  fromNewSignup ? "You've seen what we noticed and what we'll start you on. Here's what's next."
  : fromTrialExpired ? "Your trial ended. Pick up where you left off."
  : fromLapsedReadonly ? "Welcome back. Resume Vela?"
  : null; // no contextual line
```

A returning lapsed user reading "You've seen what we noticed and what we'll start you on" would be confused — they have months of data already. The `fromNewSignup` gate prevents the wrong line from appearing.

## Persistence & data lifecycle

### Pre-paywall data

All onboarding answers + the baseline scan + the generated face preview observation + the routine preview are persisted **before** the user sees the paywall:

- WatermelonDB locally (immediately).
- Supabase via background sync as soon as auth is established.

### If the user bounces at the paywall

Their data is still there. If they return within 30 days (via the same email/Apple Sign In), they re-enter the app at the paywall — *not* at onboarding. Above the paywall: *"Welcome back. Your scan and your routine are here when you're ready."* The face preview card and routine preview card are accessible from a small *"See what you started"* link.

This is a meaningful retention move on its own — many users come back days or weeks later, and the standard "redo onboarding" path of most apps loses them. Vela holds their place.

### If the user never returns

After 30 days of pre-paywall inactivity, the data is purged from Supabase server-side. Photos, scans, and onboarding answers all deleted. The user is told this in the privacy primer during sign-up: *"If you don't continue, we delete everything within 30 days."*

## Paywall integration

The paywall (file 08) gets one new contextual line at the top, dynamically rendered when the user is arriving from the new previews:

> *"You've seen what we noticed and what we'll start you on. Here's what's next."*

The line vanishes for users who arrive at the paywall through other paths (deep link, returning lapsed user, etc.) — see file 47 for those entry points.
