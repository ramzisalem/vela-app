# 44 — Experiment Mode

## Why this exists

Skincare advice on the internet is universal — *"add niacinamide,"* *"use ceramides,"* *"try a vitamin C serum."* The rules implicitly assume the user is the average. They aren't. A serum that calmed your friend's redness might do nothing for yours. The only honest way to know what works for *your* face is to test it on *your* face, with controls, over enough weeks to matter.

Experiment Mode is Vela's "AI lab partner" feature: the user picks a single change, runs it for four weeks, holds everything else steady, and sees a verdict grounded in their own data. It's an N-of-1 test on a face.

The feature does three things at once:

1. **Builds a category Vela can own.** Nobody else does personal experimentation in a face-tracking app. The looksmaxxing apps push universal advice; the diary apps just record what you ate. Experiment Mode positions Vela as the rigorous one.
2. **Increases retention durability.** A user mid-experiment is highly unlikely to churn — they're literally waiting on results.
3. **Serves the Maya persona directly.** Quantified-self users will test things constantly. They'll tell other Maya-shaped users about it.

The hardest part of this feature is the verdict. We have to be honest about what 4 weeks of data on one person can and cannot tell us. Overclaiming makes Vela a quack; underclaiming makes the feature pointless. The honest middle is *"there's a real-looking effect"* / *"it's hard to say"* / *"probably nothing"* — calibrated, hedged, plain-language. This file specifies how we do that.

## Product principles

1. **One change at a time.** The hypothesis must be a single variable. Add niacinamide. Switch SPF. Cut alcohol. We refuse experiments with multiple changes — that's not science, that's noise.
2. **Hold the routine steady.** While an experiment runs, the rest of the user's routine is locked. This protects the experiment's validity and is communicated upfront.
3. **Be honest about small samples.** Four weeks of one person's data is interesting but not definitive. The verdict copy reflects this — *always*.
4. **Never recommend the change.** Vela does not suggest adding niacinamide and then run a "test" of it. The user picks. We help them measure.
5. **Bias-aware.** We surface the placebo effect, lifestyle confounds (sleep, stress, weather), and the user's own enthusiasm bias as part of the verdict.

## What an experiment is

```ts
// src/types/experiment.ts
// Add to 02_TYPES_AND_MODELS.md.

export interface Experiment {
  id: string;
  userId: string;
  // The change being tested.
  hypothesis: ExperimentHypothesis;
  // What metric the user is trying to move.
  primaryMetric: FaceMetric;
  // Optional secondary metric (the user picks one or two).
  secondaryMetrics?: FaceMetric[];
  // Duration in weeks. Default 4, configurable to 6 or 8 by user.
  durationWeeks: 4 | 6 | 8;
  startDate: string;            // YYYY-MM-DD
  endDate: string;              // computed
  status: 'planning' | 'active' | 'completed' | 'aborted';
  // Snapshot of routine at start; locked during the experiment.
  baselineRoutineSnapshot: RoutineSnapshot;
  // The user's prediction at the start (optional).
  userPrediction?: { direction: 'up' | 'down' | 'no-change'; confidence: 1|2|3 };
  // Compliance log: did the user actually do the experimental change each day?
  complianceLog: { date: string; complied: boolean }[];
  // The verdict, generated at end.
  verdict?: ExperimentVerdict;
  createdAt: string;
  updatedAt: string;
}

export interface ExperimentHypothesis {
  // The kind of change.
  kind: 'add-product' | 'remove-product' | 'switch-product'
      | 'lifestyle-change' | 'frequency-change' | 'custom';
  // Plain-language label, user-written.
  label: string;                // ≤80 chars
  // Optional reference to a known product/treatment.
  referenceTreatmentId?: TreatmentId;
  // The action the user has committed to taking daily.
  dailyAction: string;          // ≤140 chars, user-written
}

export interface ExperimentVerdict {
  // Generated by the verdict engine + AI copy.
  effectSize: 'meaningful' | 'small' | 'unclear' | 'none' | 'inverted';
  primaryMetricDelta: number;          // points
  primaryMetricDeltaConfidence: number; // 0-1
  // The natural-aging band's expected drift over the same window.
  expectedDriftFromBand: number;
  // How much of the observed change is plausibly attributable to the experiment vs. baseline drift.
  attributableDelta: number;
  // Confounders we noticed.
  confounders: Confounder[];
  // The plain-language verdict (≤120 words).
  copy: string;
  // Whether the user complied at the threshold needed for the verdict to be valid.
  complianceRate: number;       // 0-1
  // Whether to recommend continuing, stopping, or running a new experiment.
  recommendation: 'continue' | 'stop' | 'run-again' | 'try-something-else';
  generatedAt: string;
}

export interface Confounder {
  kind: 'sleep-shift' | 'stress-shift' | 'season-shift' | 'cycle-shift'
      | 'weight-shift' | 'travel' | 'illness' | 'big-life-event';
  evidence: string;             // ≤80 chars
}
```

## The experiment flow, end-to-end

### 1. Discovery
The feature first appears as a drip-feed reveal (file 43) at week 14, eligibility:
- ≥ 8 scans
- ≥ 28 consecutive routine days
- Active subscription

Reveal copy:
> *"You've kept a routine for a month. Want to test something — a single change, four weeks, see what your face does?"*

The user can also discover it from Settings → *What's new in Vela*, or through a contextual nudge inside the routine screen if they edit a task: *"Trying something new? An experiment is the cleaner way to test it."*

- Tappable for the entire day; defaults to "Marking later" if untouched by end-of-day.
- "Marking later" entries become "No" at end-of-day automatically (we err on the strict side for compliance).
- Compliance is also computable via the routine engine if the user added the experimental change as a routine task — but we surface the prompt anyway, because the explicit yes/no is part of the experimental ritual.

#### Mid-experiment dashboard

The dashboard's primary metric chart gains an "experiment" overlay during the run: the start point is marked, the projected baseline drift band is shown (from file 36 aging band data), and the user's actual line is rendered. No verdict, just the live data.

A small persistent chip at the top of the dashboard:
> *"Experiment running · Day 12 of 28"*

Tap → opens the experiment detail screen with current data, compliance log, and notes.

#### Diary integration

Every diary entry during an experiment can be tagged as "during experiment." If the diary entry has tags that the verdict engine considers confounders (`sick`, `travel`, `big-life-event`, `stressed` for ≥ 3 entries), they surface in the verdict's confounders list.

#### Stopping early

A small *"Stop the experiment"* button in the experiment detail screen. Confirmation:

```
Stopping early?

We'll still compute a verdict — just based on the data we have.
Anything you've learned this far isn't lost.

[ Stop the experiment ]   [ Keep going ]
```

After stopping:
- `status` → `'aborted'`
- Verdict is generated with whatever data exists (with appropriate caveats about partial data).
- Routine engine resumes adaptation.


The headline is one of:
- *"It looks like there's a real improvement."* (effectSize === 'meaningful', positive direction)
- *"There's a small effect, but it's close to noise."* (effectSize === 'small')
- *"Hard to say."* (effectSize === 'unclear')
- *"We didn't see anything."* (effectSize === 'none')
- *"Things actually moved the other way."* (effectSize === 'inverted')

#### Card 2 — The chart

The primary metric over the experiment window. The user's actual line, the expected baseline drift band (from aging data), and the start/end markers. Caption: *"Your line vs. what we'd expect from drift alone."*

#### Card 3 — The math, plainly

```
What we saw:
  Your skin clarity moved +3.2 points over four weeks.
  Drift alone would have explained about +0.5.
  That leaves +2.7 attributable to the experiment.

What we hedged for:
  · Compliance was 92% (you did the daily action 26 of 28 days).
  · You logged "stressed" in your diary for two days mid-experiment.
  · Your sleep was steady (no big shifts).

Bottom line: this looks like a real, modest improvement.
```

The numbers and the bullet list of confounders are the trust anchor. Plain English, real numbers, real caveats.

#### Card 4 — Your prediction

If the user made one in Step 5:
> *"You predicted: 'Get better, pretty sure.' You were right."*

Or:
> *"You predicted: 'Get better, confident.' It came in smaller than you expected, but in the same direction."*

Or:
> *"You predicted: 'Stay the same.' It moved up — bigger than you expected."*

#### Card 5 — What's next

```
What's next?

Niacinamide was a clean win for you. You can keep it
in your routine — we'll add it back to your daily set —
or run a new experiment with something else.

[ Keep niacinamide in my routine ]
[ Run another experiment ]
[ Just go back to normal ]
```

The user's choice updates the routine engine accordingly.

#### Sharing the verdict

The verdict is shareable as a stylized card (file 13's share-cards system). The card shows:

- The headline ("It looks like there's a real improvement.")
- The before/after primary-metric delta as a single number with confidence band.

## The verdict engine

### Statistical approach

We use a deliberately simple framework — fancy doesn't help with N=28 datapoints, and pretending otherwise undermines trust.

```ts
function computeVerdict(experiment: Experiment, weeklyScans: Scan[]): ExperimentVerdict {
  // 1. Compute observed delta in primary metric.
  const startScan = weeklyScans[0];
  const endScan = weeklyScans[weeklyScans.length - 1];
  const observedDelta = endScan.primaryMetric - startScan.primaryMetric;

  // 2. Compute expected baseline drift over the same window using aging band.
  const expectedDrift = lookupAgingDrift(
    primaryMetric, user.ageDecade, user.sexAtBirth, durationWeeks
  );

  // 3. Compute attributable delta.
  const attributableDelta = observedDelta - expectedDrift.median;

  // 4. Confidence: how much of attributable delta exceeds the band's noise floor.
  // The aging band has a 10/90 percentile; the noise floor is half the band width.
  const noiseFloor = (expectedDrift.p90 - expectedDrift.p10) / 2;
  const confidence = Math.min(1, Math.max(0,
    (Math.abs(attributableDelta) - noiseFloor) / noiseFloor
  ));

  // 5. Categorize.
  let effectSize: ExperimentVerdict['effectSize'];
  if (confidence < 0.2) effectSize = 'none';
  else if (confidence < 0.4) effectSize = 'unclear';
  else if (confidence < 0.7) effectSize = 'small';
  else effectSize = 'meaningful';
  if (Math.sign(attributableDelta) !== Math.sign(experiment.userPrediction?.direction === 'up' ? 1 : -1)
      && experiment.userPrediction
      && Math.abs(attributableDelta) > noiseFloor) {
    effectSize = 'inverted';
  }

  // 6. Detect confounders from diary tags and HealthKit data during window.
  const confounders = detectConfounders(experiment, weeklyScans);

  // 7. Adjust confidence down if compliance was poor or confounders are heavy.
  if (complianceRate(experiment) < 0.7) effectSize = downgrade(effectSize);
  if (confounders.length >= 3) effectSize = downgrade(effectSize);

  // 8. Generate copy via AI.
  const copy = generateVerdictCopy(experiment, attributableDelta, effectSize, confounders);

  return { /* ... */ };
}
```

### Why this approach

- It's auditable. We can show the user the actual math.
- It's honest about noise. Small N is treated as small N.
- It uses the aging band as a population baseline, which we already have and trust.
- It respects compliance and diary-tagged confounders without trying to do causal inference.

We deliberately **do not** use techniques that pretend to more rigor than we have:
- No Bayesian priors that make weak data look strong.
- No "AI-detected effects" that bypass the math.
- No t-tests on a sample of one.

## Edge cases

- **User runs two experiments simultaneously** — not allowed. While one is active, starting another shows: *"You're already mid-experiment. We can only test one thing at a time — that's the whole point."*
- **User changes their routine mid-experiment via the routine editor** — blocked with a sheet: *"Your routine is locked while the experiment runs. Otherwise we can't tell what made the difference. Stop the experiment first?"*
- **User runs an experiment that overlaps a treatment** (already tracking tretinoin in file 34) — flagged at planning: *"You're tracking tretinoin already. Adding another change makes it harder to read. Want to wait until that timeline finishes?"* They can override.
- **Massive life event during the experiment** — if the diary tags include `big-life-event` for ≥ 3 days OR `sick` for ≥ 5 days, the verdict's confidence is downgraded automatically and the verdict copy explicitly acknowledges it: *"This was a complicated month. The data is what it is, but it's worth taking with a grain of salt."*
- **User skips multiple weekly scans** — the verdict requires a minimum of 3 scans within the window. With fewer than 3, the verdict is replaced by: *"We didn't have enough data for a clean read. The experiment didn't fail — we just didn't see enough of your face to call it. Want to try again?"*
- **Non-binary user with `combined` aging band** — the verdict math uses the wider combined band, so the noise floor is larger. The user's verdict is more conservative by default. We acknowledge this in the math card: *"We used a broad reference band, so the bar for 'real' is a little higher."*
- **HealthKit not connected** — confounder detection is limited to diary tags and self-reported compliance. We don't penalize the user for this; the verdict is computed with available data.
- **AI proxy fails to generate verdict copy** — fall back to a deterministic template that still cites the numbers and confounders. Better to ship a clean fallback than no verdict.
- **User completes the experiment and immediately wants to run the same experiment again with a 6-week duration** — allowed. We treat it as a fresh experiment; no cross-experiment data sharing.

## Privacy

- All experiment data is owned by the user account, encrypted at rest in Supabase, and included in account deletion / data export.
- The hypothesis label and daily action are user-written free text; same encryption rules as diary entries (file 37).
- The AI proxy receives: hypothesis label (truncated, sanitized), metric values, confounder list, compliance rate. It never receives photos, depth maps, or diary text.
- Verdict share cards strip all metadata; the user's photo is never included.
